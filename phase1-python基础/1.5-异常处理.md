## 核心问题：Python 为什么"滥用"异常？

在 C# 中，异常是用来处理**真正的异常情况**的——文件找不到、网络断开、权限不足。
在 Python 中，异常被用来处理**日常控制流**——这就是 EAFP 哲学。

这不是风格差异，而是**设计哲学的根本分歧**。

## 1. try/except/else/finally

### else 子句的独特作用

```python
try:
    data = load_file("config.json")
except FileNotFoundError:
    print("文件不存在，使用默认配置")
    data = DEFAULT_CONFIG
except json.JSONDecodeError as e:
    print(f"文件格式错误: {e}")
    data = DEFAULT_CONFIG
else:
    print(f"成功加载 {len(data)} 条配置")  # 只在 try 没抛异常时执行
finally:
    print("清理资源")  # 不管怎样都执行
```

**else 的作用：** 区分"成功"和"异常处理"的代码。

```python
# 不用 else 的写法：
try:
    data = load_file("config.json")
    print(f"成功加载 {len(data)} 条配置")  # 这行可能在异常处理中被误解
except FileNotFoundError:
    data = DEFAULT_CONFIG

# 用 else 的写法：
try:
    data = load_file("config.json")
except FileNotFoundError:
    data = DEFAULT_CONFIG
else:
    print(f"成功加载 {len(data)} 条配置")  # 明确：只有成功时才执行
```

**C# 有吗？** C# 没有 `else` 子句。C# 的 `try-catch-finally` 模式一般把成功逻辑放在 try 块末尾。

### 捕捉多个异常

```python
# 多个 except 块
try:
    result = risky_operation()
except ValueError:
    print("值错误")
except TypeError:
    print("类型错误")

# 一个 except 捕捉多个类型
try:
    result = risky_operation()
except (ValueError, TypeError) as e:
    print(f"值或类型错误: {e}")
```

## 2. 自定义异常

### 对比 C# 的 Exception 层级

C# 的惯例：每个场景定义一个异常类。

```csharp
public class UserNotFoundException : Exception {
    public UserNotFoundException(int userId) 
        : base($"User {userId} not found") { }
}
```

Python 同样可以自定异常，但更轻量：

```python
class UserNotFoundError(Exception):
    """用户未找到时抛出"""
    pass

def find_user(user_id: int):
    raise UserNotFoundError(f"User {user_id} not found")

try:
    find_user(42)
except UserNotFoundError as e:
    print(e)  # User 42 not found
```

**最佳实践：**

```python
# 先定义一个应用的基类异常，所有业务异常继承它
class AppError(Exception):
    """应用基类异常"""
    pass

class ConfigError(AppError):
    """配置相关错误"""
    pass

class NetworkError(AppError):
    """网络相关错误"""
    pass

# 这样上层可以：
try:
    run_app()
except AppError as e:
    # 统一处理所有业务异常
    log_error(e)
except Exception as e:
    # 处理意料之外的异常
    crash_report(e)
```

## 3. with 语句和上下文管理器

### 为什么比 try/finally 更好？

**需求：** 用完资源要释放——文件、锁、数据库连接。

```python
# 传统 try/finally
f = open("file.txt", "r")
try:
    content = f.read()
finally:
    f.close()

# with 语句（上下文管理器）
with open("file.txt", "r") as f:
    content = f.read()
# 自动关闭，不需要 finally
```

**with 语句保证：** 无论 with 块里的代码是否抛异常，退出时都会调用资源的清理方法。

### 实现自己的上下文管理器

```python
class ManagedFile:
    def __init__(self, filename, mode):
        self.filename = filename
        self.mode = mode

    def __enter__(self):
        print(f"打开文件: {self.filename}")
        self.file = open(self.filename, self.mode)
        return self.file  # 这个对象会赋给 as 后面的变量

    def __exit__(self, exc_type, exc_val, exc_tb):
        print(f"关闭文件: {self.filename}")
        self.file.close()
        # 返回 True 表示"异常已处理，不用抛出去了"
        # 返回 False 或 None 表示"让异常继续传播"
        return False

with ManagedFile("test.txt", "w") as f:
    f.write("Hello, World!")
```

### 用 `@contextmanager` 装饰器简化

```python
from contextlib import contextmanager

@contextmanager
def managed_file(filename, mode):
    print(f"打开文件: {filename}")
    try:
        f = open(filename, mode)
        yield f          # 这是 as 后面的值
    finally:
        print(f"关闭文件: {filename}")
        f.close()

with managed_file("test.txt", "w") as f:
    f.write("Hello, World!")
```

**本质：** `yield` 分开了"进入"和"退出"的逻辑。`yield` 之前的代码 = `__enter__`，之后的 = `__exit__`。

## 4. raise 和异常链

### 保留原始异常

```python
def load_config():
    try:
        with open("config.json") as f:
            return json.load(f)
    except FileNotFoundError as e:
        # 错误1：丢失原始异常
        raise ValueError("找不到配置文件")
        
        # 错误2：手动拼接，丢失栈信息
        raise ValueError(f"找不到配置文件: {e}")

        # ✅ 正确：保留原始异常的栈
        raise ValueError("找不到配置文件") from e

# 使用
try:
    load_config()
except ValueError as e:
    print(e)         # 找不到配置文件
    print(e.__cause__)  # FileNotFoundError 原始异常
```

### 重新抛出一个异常

```python
try:
    process_data(data)
except Exception as e:
    log_error(e)
    raise  # 重新抛出同一个异常，保留原有栈
```
