## 核心问题：Python 的 OOP 和 C# 有什么根本不同？

C# 的 OOP 是**类型系统驱动的**——静态检查、接口约束、编译时多态。
Python 的 OOP 是**行为驱动的**——运行时决定、鸭子类型、不需要事先声明"我会什么"。

这不是"弱化版"，而是完全不同的思路。

## 1. class & `__init__`：创建对象的两个步骤

### `__new__` vs `__init__`

```python
class Person:
    # __new__：第一步——分配内存，创建实例（极少重写）
    def __new__(cls, *args, **kwargs):
        print(f"1. 创建 {cls.__name__} 实例")
        instance = super().__new__(cls)
        return instance

    # __init__：第二步——初始化实例属性
    def __init__(self, name: str, age: int):
        print(f"2. 初始化实例")
        self.name = name
        self.age = age

p = Person("Alice", 30)
# 1. 创建 Person 实例
# 2. 初始化实例
```

**分工：**
- `__new__` 是**类方法**，负责创建对象（分配内存）
- `__init__` 是**实例方法**，负责初始化对象（设置属性）

**99% 的情况你只需要 `__init__`。** 只有当你想控制对象创建过程（比如单例模式、缓存）时才重写 `__new__`。

**对比 C#：** C# 的构造函数同时做了创建和初始化两步。Python 分开，给了你中间插手的空间。

```csharp
// C#：一步到位
class Person {
    public Person(string name, int age) { ... }
}
```

### self 是什么？

```python
class Dog:
    def __init__(self, name):
        self.name = name

    def bark(self):
        return f"{self.name} says woof!"

# self 就是"调用方法的那个实例"
dog = Dog("Buddy")
dog.bark()        # Python 自动把 dog 传给 self

# 等价写法（手动传 self）：
Dog.bark(dog)     # same thing
```

**为什么 self 必须显式写？**
- Python 没有 `this` 关键字
- 方法本质上就是函数，`self` 只是第一个参数
- 显式写 self 让你清楚哪个变量是实例属性，哪个是局部变量

## 2. `@property`：核心理解

### 为什么 Python 不需要 C# 的 get/set 方法？

C# 中，你可能这样写：

```csharp
private string _name;
public string Name {
    get { return _name; }
    set { _name = value; }
}
```

后来 C# 3.0 给你自动属性：
```csharp
public string Name { get; set; }
```

Python 的哲学：**不要提前封装。先用普通属性，需要时再加逻辑——而且调用的地方不用改代码。**

```python
class User:
    def __init__(self, name: str):
        self._name = name  # 普通属性

    @property
    def name(self):         # 假装是属性，实际上是方法
        return self._name.strip().title()

    @name.setter
    def name(self, value):
        if not value.strip():
            raise ValueError("名字不能为空")
        self._name = value.strip()

# 调用方——和普通属性一模一样！
user = User("  alice  ")
print(user.name)           # "Alice"（自动去除空格并首字母大写）
user.name = "  bob  "      # 走 setter
```

**本质：** `@property` 是**Get 方法伪装成字段**。

| 步骤 | Python | C# |
|------|--------|-----|
| 初始 | `self.name = x` | `public string Name { get; set; }` |
| 需要封装时 | 加 `@property` | 加 `private _name` + 读写属性 |

**Python 思路：** 先简单，复杂了再升级。C# 思路：先规划好，一开始就用属性。

### property 如何实现"看起来像属性"？

```python
# property 是一个类，不是语法！@property 是装饰器语法糖
# 等价于：
name = property(get_name, set_name, del_name, doc_string)
```

## 3. `@classmethod` vs `@staticmethod`

```python
class Date:
    def __init__(self, year: int, month: int, day: int):
        self.year = year
        self.month = month
        self.day = day

    # 实例方法：需要 self（实例）
    def display(self):
        return f"{self.year}-{self.month:02d}-{self.day:02d}"

    # 类方法：需要 cls（类本身），可以访问类属性
    @classmethod
    def from_string(cls, date_str: str):
        parts = date_str.split("-")
        return cls(int(parts[0]), int(parts[1]), int(parts[2]))

    # 静态方法：不需要 self 或 cls，就是一个函数放在类命名空间里
    @staticmethod
    def is_valid(date_str: str):
        try:
            parts = date_str.split("-")
            y, m, d = int(parts[0]), int(parts[1]), int(parts[2])
            return 1 <= m <= 12 and 1 <= d <= 31
        except:
            return False

# 使用
d = Date.from_string("2024-03-15")  # 类方法：工厂模式
print(Date.is_valid("abc"))         # 静态方法：验证工具
print(d.display())                  # 实例方法
```

**三种方法的本质区别：**

| 类型 | 第一个参数 | 用途 | C# 对应 |
|------|-----------|------|---------|
| 实例方法 | `self` | 操作实例 | 普通方法 |
| `@classmethod` | `cls` | 工厂方法、替代构造器 | 静态方法（但能访问泛型类型参数） |
| `@staticmethod` | 无 | 工具函数 | 静态方法 |

**理解角度：** `@classmethod` 和 `@staticmethod` 都是在创造**被类调用的函数**，不是实例调用的。区别在于 `@classmethod` 知道自己是哪个类（可以被子类正确调用），`@staticmethod` 不知道。

```python
class Parent:
    @classmethod
    def create(cls):
        return cls()  # 返回的是子类的实例！

    @staticmethod
    def info():
        return "This is a thing"

class Child(Parent):
    pass

print(Child.create().__class__.__name__)  # "Child" —— 正确
print(Child.info())                       # "This is a thing"
```

## 4. 特殊方法（Magic Methods）

### 完整体系

Python 的特殊方法是**协议**（Protocol）的基础。你实现特殊方法，你的对象就"支持"了某些操作。

```python
class Person:
    def __init__(self, name: str, age: int):
        self.name = name
        self.age = age

    # 交互输出：print() 调 __str__，repr() 调 __repr__
    def __repr__(self):
        return f"Person('{self.name}', {self.age})"
    
    def __str__(self):
        return f"{self.name} ({self.age})"

    # 相等与哈希（必须同时定义）
    def __eq__(self, other):
        if not isinstance(other, Person):
            return NotImplemented
        return self.name == other.name and self.age == other.age
    
    def __hash__(self):
        return hash((self.name, self.age))

    # 让对象可调用
    def __call__(self, greeting="Hello"):
        return f"{greeting}, I'm {self.name}!"

    # 上下文管理器（with 语句）
    def __enter__(self):
        print("进入上下文")
        return self
    
    def __exit__(self, exc_type, exc_val, exc_tb):
        print("退出上下文")

# 使用
p = Person("Alice", 30)
print(p)          # Alice (30)
print(repr(p))    # Person('Alice', 30)

print(p())         # Hello, I'm Alice!
print(p("Hi"))    # Hi, I'm Alice!

people = {p: "friend"}   # 因为实现了 __hash__，可以用作字典键

with p:
    print("在上下文里")
```

**对比 C#：** C# 没有"特殊方法"体系，而是通过接口（IEquatable、IDisposable、INotifyPropertyChanged）和运算符重载来实现类似功能。

| 方法 | 触发 | C# 对应 |
|------|------|---------|
| `__init__` | `obj = Class()` | 构造函数 |
| `__str__` | `print(obj)` | `ToString()` |
| `__repr__` | `repr(obj)` | —（通常和 ToString 一样） |
| `__eq__` | `obj == other` | `Equals()` |
| `__hash__` | `hash(obj)` | `GetHashCode()` |
| `__call__` | `obj()` | 无（C# 没有把对象当函数调用的机制）|
| `__enter__/__exit__` | `with obj:` | `IDisposable` |
| `__len__` | `len(obj)` | `Count` 属性 |
| `__getitem__` | `obj[key]` | 索引器 `this[]` |
| `__iter__` | `for x in obj:` | `GetEnumerator()` |

### 为什么 `__hash__` 和 `__eq__` 必须同时定义？

```python
class Person:
    def __init__(self, name):
        self.name = name
    
    def __eq__(self, other):
        return isinstance(other, Person) and self.name == other.name
    
    # ❌ 没定义 __hash__，Python 自动设为 None
    # 导致 Person 实例不可哈希，不能放进 set 或 dict 的键

p = Person("Alice")
d = {p: "data"}  # TypeError: unhashable type: 'Person'
```

**原因（和 C# 完全相同）：** 哈希表要求：如果 a == b，那么 hash(a) == hash(b)。如果你自定义了相等逻辑，却不自定义哈希，两个相等的对象可能哈希值不同，导致哈希表损坏。

```python
# ✅ 正确做法
class Person:
    def __init__(self, name):
        self.name = name
    
    def __eq__(self, other):
        return isinstance(other, Person) and self.name == other.name
    
    def __hash__(self):
        return hash(self.name)  # 和 __eq__ 使用相同的判定字段
```

## 5. 继承与 MRO

### super() 的调用链

```python
class A:
    def say(self):
        print("A")

class B(A):
    def say(self):
        print("B -> ", end="")
        super().say()

class C(A):
    def say(self):
        print("C -> ", end="")
        super().say()

class D(B, C):
    def say(self):
        print("D -> ", end="")
        super().say()

d = D()
d.say()     # D -> B -> C -> A
print(D.__mro__)  # D -> B -> C -> A -> object
```

**MRO（Method Resolution Order）** 由 C3 线性化算法确定，保证了三个特性：
1. 子类优先于父类
2. 父类声明顺序（B 先于 C）
3. 单调性

**为什么不是 D -> B -> A -> C？** 那样 C 的 say() 会被跳过，违背了 C 也被继承的事实。C3 线性化确保每个类只出现一次，且所有父类都被考虑到。

### super() 不要省略参数

```python
# Python 3 可以简写为 super()，等价于 super(D, self)
# 但理解它的含义很重要：
super(D, self).say()  # 在 MRO 中找 D 的下一个类，调用它的 say()
```

## 6. ABC 与抽象类

### 对比 C# 的 abstract class 和 interface

```python
from abc import ABC, abstractmethod

# ABC = 抽象基类（类似 abstract class）
class Animal(ABC):
    @abstractmethod
    def speak(self) -> str:
        pass

    def breathe(self):
        return "breathing..."  # 可以有具体方法

class Dog(Animal):
    def speak(self):
        return "Woof!"

# a = Animal()  # ❌ TypeError：不能实例化抽象类

# 对比 C#：
# abstract class Animal {
#     public abstract string Speak();
#     public string Breathe() => "breathing...";
# }
```

### ABC vs Duck Typing

ABC 是**强约束**："你必须是 Animal 的子类才能用"。
Duck Typing 是**弱约束**："你会 speak() 就行"。

```python
# 强约束（ABC）
def process_animal(animal: Animal):
    print(animal.speak())   # 保证有 speak()

# 弱约束（Duck Typing）
def process_speaker(speaker):
    print(speaker.speak())  # 运行时才知道有没有 speak()
```

Python 3.8+ 的 Protocol（见下文）提供了两种思路的中间地带。

## 7. `@dataclass`

### 自动生成了什么？

```python
from dataclasses import dataclass

@dataclass
class Point:
    x: float
    y: float

# @dataclass 自动生成：
# 1. __init__：def __init__(self, x: float, y: float)
# 2. __repr__：def __repr__(self) -> "Point(x=1.0, y=2.0)"
# 3. __eq__：def __eq__(self, other) -> bool

p1 = Point(1, 2)
p2 = Point(1, 2)
print(p1)          # Point(x=1, y=2)       —— 来自自动 __repr__
print(p1 == p2)    # True                  —— 来自自动 __eq__
```

**对比 C# 的 record：**

```csharp
// C# 9+ record
record Point(float X, float Y);
// 自动生成：构造器、ToString、Equals、GetHashCode、Deconstruct、with 表达式
```

| 特性 | Python dataclass | C# record |
|------|-----------------|-----------|
| 自动 `__init__` | ✅ | ✅ |
| 自动 `ToString` / `__repr__` | ✅ | ✅ |
| 自动 `Equals` / `__eq__` | ✅ | ✅ |
| 自动 `GetHashCode` / `__hash__` | 可选（`frozen=True`）| ✅ |
| 值相等 | 默认（可配置）| 默认 |
| 不可变性 | `frozen=True` | `record class`(1) / `record struct`(2) |
| 克隆/with 表达式 | `replace(p, x=3)` | `p with { X = 3 }` |

### dataclass 进阶

```python
from dataclasses import dataclass, field

@dataclass(frozen=True)  # 不可变
class Config:
    host: str
    port: int = 8080      # 默认值

@dataclass
class Student:
    name: str
    grades: list = field(default_factory=list)  # 可变默认值的正确写法！
    # ❌ 不要写：grades: list = [] —— 所有实例共享同一个列表
```

## 8. Mixin

### 为什么 Python 需要这个模式？

**C# 只有单继承 + 接口。** 接口定义契约，但不提供实现。

**Python 支持多继承。** Mixin 就是利用多继承来复用行为的小类。

```python
class JSONMixin:
    def to_json(self):
        import json
        return json.dumps(self.__dict__)

class LogMixin:
    def log(self, message):
        print(f"[{self.__class__.__name__}] {message}")

class User(JSONMixin, LogMixin):
    def __init__(self, name):
        self.name = name
        self.log(f"创建用户: {name}")

user = User("Alice")
print(user.to_json())   # {"name": "Alice"}
```

**Mixin 的设计原则：**
- 只做一件事
- 不维护自己的状态（没有 `__init__`）
- 不依赖特定的子类（只依赖自己的方法）

**C# 里怎么模拟？** 用扩展方法（扩展方法只能访问 public API，不如 Python Mixin 强大）或默认接口方法（C# 8+）。

## 9. `__slots__`

### 为什么能省内存？

```python
class RegularUser:
    def __init__(self, name):
        self.name = name

class SlottedUser:
    __slots__ = ("name",)  # 只允许 name 属性
    def __init__(self, name):
        self.name = name

u1 = RegularUser("Alice")
u2 = SlottedUser("Alice")

u1.age = 30      # ✅ 普通类：可以动态添加属性
u2.age = 30      # ❌ AttributeError：SlottedUser 没有 age

# __slots__ 省内存的原因：
# 普通实例用 dict 存属性（每个实例一个字典，有哈希表开销）
# Slotted 实例用固定大小的数组（类似 C 的 struct）
```

**使用场景：** 创建大量实例时（百万级），`__slots__` 能显著减少内存。

**限制了什么：** 不能动态添加属性，不能使用 `__dict__`，多继承时有限制。

## 10. Protocol（3.8+）

### "结构子类型" vs "名义子类型"

```python
from typing import Protocol

# 名义子类型（C# 的接口）：
#   "你声明你实现了我，你就是我的子类型"
# 结构子类型（Python Protocol）：
#   "你实现了我的方法，你就是我的子类型"

class Speaker(Protocol):
    def speak(self) -> str:
        ...

class Dog:
    def speak(self):
        return "Woof!"

class Cat:
    def speak(self):
        return "Meow!"

# Dog 和 Cat 没有声明继承 Speaker，但符合 Speaker 的结构
def greet(speaker: Speaker):
    print(speaker.speak())

greet(Dog())   # 类型检查器接受（虽然 Dog 没写 "implements Speaker"）
greet(Cat())   # ✅

class Robot:
    def beep(self):
        return "Beep!"

greet(Robot())  # ❌ 类型检查器拒绝（Robot 没有 speak()）
```

**本质：** Protocol = "静态鸭子类型"。在类型检查层面做鸭子类型的检查，不需要运行时开销。

**对比 C#：** C# 的接口是**名义子类型**——必须显式声明 `: ISpeaker`。Python 的 Protocol 是**结构子类型**——只要结构匹配就算。Protocol 更像 TypeScript 的 interface。

### 练习 5：思考题——property vs 直接属性

```python
class Temperature:
    def __init__(self, celsius: float):
        self._celsius = celsius

    @property
    def celsius(self):
        return self._celsius

    @celsius.setter
    def celsius(self, value):
        self._celsius = value

    @property
    def fahrenheit(self):
        return self._celsius * 9/5 + 32
```

`celsius` 的 setter 只是直接赋值，什么额外逻辑都没加。那为什么不直接用 `self.celsius = value` 而要用 property？换句话说，先写普通属性，等需要了再加 property 就行——现在这个 setter 纯属多余。你能理解这个观点吗？反过来，什么情况下你**应该**从一开始就用 property？
