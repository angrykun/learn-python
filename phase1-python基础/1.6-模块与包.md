## 核心问题：C# 的 using 和 Python 的 import 有什么本质区别？

**C# 的 using：** 告诉编译器「我要用这个命名空间里的类型」。编译时解析。
**Python 的 import：** 在运行时**执行一个文件**，把它的名字空间加载到当前名字空间。

这个差异解释了一切。

## 1. import 机制：到底做了什么？

```python
# 当你写：
import math

# Python 实际上做了：
# 1. 在 sys.path 中查找 math.py
# 2. 把 math.py 编译为字节码（如果是首次加载）
# 3. 执行 math.py 中的代码
# 4. 把生成的模块对象存入 sys.modules（缓存）
# 5. 在当前名字空间创建一个变量 math，指向该模块对象

print(math.pi)      # 3.141592653589793
print(math.sqrt)    # <built-in function sqrt>
```

### 三种导入方式

```python
# 方式1：导入模块，通过模块名访问
import math
print(math.pi)

# 方式2：导入模块中的特定名称（常用）
from math import pi, sqrt
print(pi)                     # 直接用，不需要 math.
print(sqrt(4))                # 2.0

# 方式3：导入模块中的所有名称（不推荐）
from math import *            # 可能覆盖你已有的变量！
pi = "not pi"                 # 覆盖了 math 的 pi
```

**为什么不推荐 `from math import *`：** 你不知道导入了什么名字，可能意外覆盖已有的变量。在大型项目中这是 bug 的来源。

## 2. `if __name__ == '__main__'`

### 这个模式为什么如此重要？

```python
# utils.py
def greet(name):
    return f"Hello, {name}!"

def add(a, b):
    return a + b

# 只有在直接运行这个文件时才执行
if __name__ == "__main__":
    print(greet("World"))
    print(f"1 + 2 = {add(1, 2)}")
```

**解释：**

- 每个 Python 文件都有内置变量 `__name__`
- 当文件**直接运行**时，`__name__` 被设置为 `"__main__"`
- 当文件**被导入**时，`__name__` 被设置为文件名（不含 .py）

```python
# 直接运行 python utils.py
# __name__ == "__main__" → 执行测试代码

# 在其他文件中 import utils
# __name__ == "utils" → 不执行测试代码
```

**对比 C#：** C# 有 `Main()` 方法作为程序入口。Python 的文件既可以当模块导入，也可以当脚本运行，`if __name__` 模式就是区分这两种场景的。

## 3. 包结构

### 用目录组织模块

```
myproject/
  __init__.py         # 包标识文件
  main.py
  utils/
    __init__.py       # 可以是空文件
    helpers.py
    parsers.py
  models/
    __init__.py
    user.py
    post.py
```

```python
# 包内引用
from utils.helpers import format_date
from models.user import User

# __init__.py 可以控制 `from package import *` 导出的内容
# __init__.py 也可以做初始化（很少需要）
```

### `__init__.py` 的作用

1. **标记目录为包**（Python 3.3+ 即使没有 __init__.py，目录也是隐式包，但显式写是惯例）
2. **控制包的公共接口**
3. **提前加载频繁使用的子模块**

```python
# models/__init__.py
from .user import User      # 让外部可以直接 from models import User
from .post import Post
```

## 4. sys.path 和模块搜索

### Python 从哪里找模块？

```python
import sys
print(sys.path)
# 通常是：
# 1. 当前脚本所在目录
# 2. PYTHONPATH 环境变量中的路径
# 3. Python 安装目录的标准库
# 4. site-packages（第三方包）
```

```python
# 添加搜索路径（临时）
import sys
sys.path.append("/path/to/my/modules")
import my_module

# 长期方案：设置 PYTHONPATH 环境变量
# export PYTHONPATH="/path/to/my/modules:$PYTHONPATH"
```

### 对比 C# 的 using

| C# | Python |
|----|--------|
| `using System.Collections;` | `import sys` |
| 编译时解析 | 运行时执行 |
| 添加引用（.csproj/NuGet） | `pip install` |
| 命名空间层次 | 目录 + `__init__.py` |
| 无缓存 | `sys.modules` 缓存已加载模块 |

**关键差异：** C# 的 using 只影响**编译时的名称解析**。Python 的 import 是**运行时的代码执行**——每 import 一个模块，就是运行了一个文件。这也是为什么 Python 的 import 可能有副作用（模块级别的代码）。
